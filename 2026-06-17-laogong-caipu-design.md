# 老公菜谱 H5 MVP 设计

## 背景

「老公菜谱」第一版是一个本地自用的手机 H5 工具，用来把小红书分享内容抓取、解析并保存成可复用的做菜攻略。项目重点不是做一个通用公开菜谱平台，而是服务一个家庭场景：从小红书收集菜谱，交给 AI 结构化，保存到本地数据库，之后做菜时查看步骤，并记录老婆评价和老公下次改进事项。

## 目标

- 支持粘贴小红书分享文本，识别其中的 `xhslink.com` 短链和分享文案。
- 后端尝试展开短链并抓取页面内容。
- 抓取失败或内容不足时，允许用户手动补充正文，继续进入 AI 解析。
- 使用 DeepSeek 作为第一版 AI 解析引擎，把原始内容转换为结构化菜谱。
- 使用 SQLite 在本地保存菜谱、标签、步骤、食材和做过记录。
- 提供手机优先的 H5 页面，用于导入、确认、浏览、筛选、查看和标记做过。

## 非目标

- 第一版不做登录、账号、多用户、权限系统。
- 第一版不部署公网，不处理公网安全、分享权限或云数据库。
- 第一版不承诺小红书抓取 100% 成功，不做复杂登录态、Cookie 管理或反爬绕过。
- 第一版不做完整桌面端适配，只保证宽屏下基本可用。
- 第一版不做复杂分类后台，分类和标签以 AI 建议加手动编辑为主。

## 技术选型

- 应用框架：Next.js 全栈应用。
- 前端：React、TypeScript、Tailwind CSS。
- 动画：Motion / Framer Motion，用于页面过渡、卡片入场、底部 Sheet、Toast 和反馈动画。
- 后端：Next.js Route Handlers。
- 数据库：SQLite，本地文件数据库。
- AI：DeepSeek，API key 和模型名通过 `.env` 配置。
- 状态管理：第一版优先使用 React 本地状态和服务端 API；如列表缓存需求变复杂，再引入轻量数据请求缓存方案。

## 手机端体验

第一版只考虑手机 H5 主体验。布局以 375px 左右宽度优先，使用底部导航、竖向卡片、底部弹窗和单手可操作的按钮位置。

视觉风格延续用户指定方向：

- 现代极简、温馨治愈、女性友好。
- 主色：暖珊瑚粉 `#FF6B6B`。
- 辅色：奶油杏 `#FFE4D6`。
- 背景：奶油白 `#FFF9F5`。
- 卡片圆角 16px，按钮圆角 24px。
- 使用柔和弥散阴影。
- 标题字重 600，正文字重 400。

动画规范迁移到 React 生态：

- 页面进入：透明度 0 到 1，纵向位移 20px 到 0，约 400ms。
- 页面离开：透明度 1 到 0，纵向位移 0 到 -20px，约 200ms。
- 卡片列表使用 stagger 入场，按索引递增延迟。
- 底部 Sheet 从底部滑入和滑出。
- 保存成功、评价提交使用 Toast；评价提交可使用 canvas-confetti。
- 加载状态使用骨架屏和脉冲动画。

## 页面与流程

### 导入页

首页即导入入口。用户粘贴小红书分享文本，例如：

```text
5分钟就可以搞定！超级下饭的丝瓜炒蛋 丝瓜炒蛋可以算... http://xhslink.com/o/smiaxnsR3c
复制后打开【小红书】查看笔记！
```

页面提供一个大输入框和主按钮“开始抓取”。提交后展示分阶段状态：

- 正在识别链接。
- 正在抓取内容。
- 正在整理成菜谱。

### 解析确认页

AI 返回结构化菜谱后进入确认页。用户可以查看并修改：

- 菜名。
- 主分类。
- 标签。
- 食材。
- 调料。
- 步骤。
- 烹饪时间。
- 难度。
- 小贴士。

如果抓取内容不足，页面提示“内容有点少，可以补充原文”，并提供正文粘贴区。用户补充后可以重新解析。

### 菜谱列表页

展示已保存菜谱卡片。卡片包含：

- 封面图。
- 菜名。
- 主分类。
- 标签摘要。
- 做过状态。
- 老婆评价摘要。

列表支持搜索菜名，并按主分类或标签筛选。第一版筛选保持轻量，不做复杂管理后台。

### 菜谱详情页

详情页用于做菜时查看，信息优先级为：

- 食材与调料。
- 步骤。
- 小贴士。
- 做过复盘。

底部固定“标记做过”按钮。点击后打开底部 Sheet，填写：

- 老婆评价。
- 老公下次改进事项。
- 我的备注。

提交后增加做过次数，并新增一条做过记录。

### 分类页

底部导航包含轻量分类入口。分类页第一版用于按主分类和标签快速筛选菜谱，不承担完整分类管理职责。

## 后端模块

### sourceParser

负责从用户粘贴文本中识别来源、提取短链和保留分享文案。第一版重点支持 `xhslink.com`，并保留兼容通用 URL 的扩展点。

输出内容包括：

- sourcePlatform。
- sourceUrl。
- shareText。
- normalizedInput。

### crawler

负责展开短链、抓取页面 HTML，并尝试提取原始内容：

- 最终跳转 URL。
- 页面标题。
- meta 描述。
- 可见正文片段。
- 图片 URL。
- 结构化数据片段。

第一版不做复杂登录态、Cookie 管理和反爬绕过。抓取失败时返回明确失败原因，而不是让导入流程直接中断。

### aiRecipeParser

负责调用 AI 引擎，把分享文案、抓取内容和手动补充正文合并后解析为统一菜谱结构。

业务层只调用统一方法：

```ts
parseRecipeFromContent(input)
```

第一版默认实现为 DeepSeek。API key 使用 `DEEPSEEK_API_KEY`，模型名使用环境变量配置。保留 Mock 实现，用于本地没有 key 或测试时使用。

### recipeRepository

负责 SQLite 读写，封装菜谱、食材、步骤、标签、导入记录和做过记录的持久化。

### importService

负责串起完整导入流程：

1. 解析输入。
2. 尝试抓取。
3. 合并内容。
4. 调用 AI 解析。
5. 校验 AI 输出。
6. 返回确认页所需数据。
7. 用户确认后保存。

同时记录每一步状态和失败原因，方便前端展示和后续调试。

## 小红书抓取策略

第一版以小红书分享文本作为核心输入，而不是只支持裸 URL。

处理策略：

1. 从整段文本中提取 `xhslink.com` 短链。
2. 保留短链前面的分享文案。
3. 服务端请求短链，获取最终跳转地址。
4. 尝试抓取最终页面的标题、meta 描述、正文片段、图片和结构化数据。
5. 如果抓取结果太少，就把分享文案作为主要输入交给 AI。
6. 如果 AI 仍然判断信息不足，前端提示用户补充原文后重新解析。

失败原因需要结构化返回：

- 短链无效。
- 页面需要登录。
- 页面内容为空。
- 网络请求失败。
- AI 解析失败。
- AI 输出格式不合法。

## AI 解析结构

AI 输入包含：

- 来源平台。
- 来源链接。
- 分享文案。
- 抓取到的标题。
- 抓取到的正文。
- 抓取到的图片。
- 用户手动补充正文。

AI 输出必须是固定 JSON 结构：

```json
{
  "name": "丝瓜炒蛋",
  "mainCategory": "家常菜",
  "tags": ["下饭", "快手菜"],
  "ingredients": [
    { "name": "丝瓜", "amount": "1根", "type": "ingredient" }
  ],
  "seasonings": [
    { "name": "盐", "amount": "适量", "type": "seasoning" }
  ],
  "steps": [
    { "order": 1, "text": "丝瓜去皮切块。" }
  ],
  "cookTimeMinutes": 5,
  "difficulty": "easy",
  "tips": "鸡蛋先炒熟盛出，最后回锅。",
  "confidence": 0.82,
  "missingFields": []
}
```

后端校验规则：

- 返回内容必须是合法 JSON。
- 必须有菜名。
- 至少有一个步骤。
- 食材、调料、标签必须是数组。
- 空字段使用空字符串、空数组或 `null`，避免字段缺失。
- 如果可信度低，前端提示用户确认或补充正文。

## 数据模型

### recipes

保存菜谱主信息：

- id。
- name。
- main_category。
- source_platform。
- source_url。
- original_title。
- share_text。
- cover_image_url。
- cook_time_minutes。
- difficulty。
- tips。
- cooked_count。
- created_at。
- updated_at。

### recipe_ingredients

保存食材和调料：

- id。
- recipe_id。
- name。
- amount。
- type，取值为 `ingredient` 或 `seasoning`。
- sort_order。

### recipe_steps

保存步骤：

- id。
- recipe_id。
- step_order。
- text。
- image_url。

### recipe_tags

保存自由标签：

- id。
- recipe_id。
- tag。

### cooking_logs

保存每次做过的复盘记录：

- id。
- recipe_id。
- cooked_at。
- wife_feedback。
- husband_improvement_notes。
- notes。
- created_at。

### imports

保存导入记录：

- id。
- raw_input。
- source_platform。
- source_url。
- share_text。
- final_url。
- crawl_status。
- crawl_error。
- ai_status。
- ai_error。
- parsed_json。
- created_recipe_id。
- created_at。

## 错误处理

错误处理原则是不中断用户流程，尽量给出下一步操作。

- 短链提取失败：提示用户检查分享文本，或手动粘贴原文。
- 抓取失败：保留分享文案，仍可进入 AI 解析。
- 内容太少：提示用户补充正文后重新解析。
- AI 解析失败：显示失败原因，可重试。
- AI 输出不合法：显示结构化失败，可重试或补充正文。
- 保存失败：保留解析结果，避免用户重新粘贴。

## 测试策略

第一版重点覆盖导入链路和数据持久化：

- 小红书分享文本能提取短链和分享文案。
- 短链展开失败时仍保留分享文案。
- AI mock 输出能通过 schema 校验。
- 非法 AI 输出会返回清晰错误。
- 菜谱、步骤、标签、食材和做过记录能正确写入 SQLite。
- 导入流程 API 能返回清晰状态。
- 手机端关键流程可操作：导入、确认、保存、列表、详情、标记做过。

## 验收标准

- 用户能在手机 H5 粘贴小红书分享文本并启动导入。
- 系统能识别 `xhslink.com` 短链和分享文案。
- 后端能尝试抓取短链页面，并在失败时保留可继续解析的输入。
- DeepSeek 解析结果能以结构化菜谱形式展示给用户确认。
- 用户能保存菜谱到 SQLite。
- 用户能在列表页找到已保存菜谱。
- 用户能进入详情页查看食材、调料、步骤和小贴士。
- 用户能标记菜谱做过，并填写老婆评价、老公下次改进事项和备注。
- 做过次数和做过记录能正确保存。
