# 老公菜谱家庭门禁、云端共享与食材图加载优化设计

日期：2026-07-19

状态：待书面复核（整体设计已获用户口头确认）

## 1. 背景与结论

当前菜谱并不是分别保存在两台手机上。SQLite 位于运行 Next.js 的服务器，两台手机只要访问同一个服务地址，就会共享菜谱、收藏、做菜次数和复盘记录。

现有风险来自运行边界，而不是 SQLite 本身：

- 服务仍依赖当前运行它的单台机器和本地磁盘。
- 项目没有登录、家庭权限或 API 门禁，不能直接暴露到公网。
- 做菜进度、计时器和导入草稿使用浏览器 `sessionStorage`，只存在当前设备。
- 已打开的另一台手机不会实时收到变化，需要刷新或重新进入页面。
- AI 食材图片虽然可以缓存，但原始输出远大于实际头像尺寸，首次生成期间也缺少清晰反馈。

本轮采用以下已确认方案：

1. 把应用部署为用户云服务器上的单实例 Next.js 服务。
2. 继续使用持久化 SQLite，不迁移 Postgres，也不引入多用户表。
3. 两个人共用一个家庭密码和同一个家庭数据空间。
4. 新增应用内家庭解锁页和 30 天安全会话。
5. 菜谱、收藏、做菜次数和复盘共享；做菜中的临时状态继续留在当前手机。
6. 为食材图片增加骨架状态、服务端小图压缩和稳定缓存。
7. 将“全部勾选”改为可再次点击取消全选。

## 2. 目标与非目标

### 2.1 目标

- 两台手机通过同一个 HTTPS 地址可靠访问同一份家庭菜谱数据。
- 未输入家庭密码的访问者不能读取或修改页面、API、菜谱和生成图片。
- 家庭密码不进入客户端代码、数据库、日志或 Git。
- 云服务器重启、应用升级后 SQLite 和生成图片缓存仍然保留。
- 食材生图时用户能明确看到“正在生成”，生成完成后快速加载小图。
- 在低并发家庭场景中保持部署与维护成本最低。

### 2.2 非目标

- 不建立老公、老婆两个独立账号。
- 不记录操作人身份，不实现个人收藏或角色权限。
- 不实现 WebSocket、SSE 或跨设备实时协同编辑。
- 不同步进行中的步骤、计时器、语音状态、备料勾选和未完成导入草稿。
- 不迁移到 Supabase、Turso 或 Postgres。
- 不在本轮未经授权连接或修改用户的云服务器。

## 3. 运行架构

### 3.1 单实例服务

- 云服务器只运行一个本项目 Next.js 应用实例。
- Hermes 与本项目使用不同进程或容器及不同内部端口，互不共享运行目录。
- Nginx 或 Caddy 负责公网域名、HTTPS 证书和反向代理。
- Next.js 只监听内部端口；数据库端口不对公网开放，因为 SQLite 没有独立数据库服务。
- 健康检查只验证应用可响应，不返回环境变量、路径或数据库内容。

单实例是本设计的硬约束。若以后需要水平扩容为多个实例，应先迁移到中心数据库和对象存储，不能让多个实例各自持有 SQLite 与图片目录。

### 3.2 持久化目录

运行时数据统一放在服务器持久化目录，例如：

```text
/srv/laogong-caipu/data/laogong-caipu.sqlite
/srv/laogong-caipu/data/generated/ingredients/v2/
```

- `DATABASE_PATH` 指向持久化 SQLite 文件。
- 食材图缓存根目录位于同一个持久化 `data` 目录。
- 若使用容器，宿主机该目录必须挂载到容器，不能只写在容器可写层。
- 运行应用的系统用户只获得该目录的读写权限。

### 3.3 SQLite 运行参数

连接初始化时启用：

- `PRAGMA foreign_keys = ON`
- `PRAGMA journal_mode = WAL`
- `PRAGMA busy_timeout = 5000`

所有跨表写入继续使用事务。两人低频使用时，WAL 与有限写入等待足以处理收藏、导入和复盘偶尔同时发生的情况。

## 4. 家庭门禁

### 4.1 页面与路由

新增中国风家庭解锁页，路径固定为 `/unlock`：

- 显示产品名称、简短家庭文案、密码输入框和 shadcn/ui `Button`。
- 复用现有中文字体、色彩 token、圆角与 Apple 式按压反馈。
- 输入框支持显示/隐藏密码，但默认隐藏。
- 登录成功返回最初请求的同源站内页面；无安全返回地址时进入首页。
- 首页菜单提供“退出家庭”操作。

以下资源无需会话即可访问：

- `/unlock`
- 登录和退出接口
- Next.js 必需的静态资源、字体、图标和健康检查

其余页面、菜谱 API、导入 API、收藏 API、复盘 API、食材图生成 API和食材图读取 API全部受门禁保护。

### 4.2 密码存储与校验

- 不保存明文密码。
- 服务器环境变量保存带随机盐的 scrypt 密码摘要 `FAMILY_PASSWORD_HASH`。
- 项目提供本地命令生成摘要；命令只输出摘要，不写入仓库文件。
- 登录接口在服务端使用固定参数的 scrypt 重新派生并以恒定时间比较。
- 密码错误统一返回“家庭密码不正确”，不暴露配置、摘要或内部异常。
- 未配置或格式错误时，服务端拒绝登录并记录不含秘密值的配置错误。

### 4.3 会话 Cookie

登录成功后签发自包含、带到期时间的 HMAC-SHA256 会话令牌：

- `HttpOnly`
- 生产环境 `Secure`
- `SameSite=Lax`
- `Path=/`
- `Max-Age=30 天`

签名密钥来自独立环境变量 `FAMILY_SESSION_SECRET`，不得复用家庭密码或 AI Key。密钥至少 32 字节随机值。

中间件验证令牌结构、签名和到期时间：

- 页面未登录时重定向到 `/unlock`，并携带经过白名单校验的站内返回路径。
- API 未登录时返回 JSON `401`，不返回 HTML 登录页。
- 会话过期、损坏或密钥轮换后均视为未登录。
- 退出接口清除 Cookie；退出后浏览器后退也不能重新访问受保护数据。

### 4.4 登录防护

- 登录接口校验同源 `Origin`，拒绝异常跨站请求。
- 同一来源连续失败达到 5 次后暂停 15 分钟；成功登录后清除失败计数。
- 限流状态只需覆盖当前单实例进程，并设置容量上限和过期清理，避免内存无限增长。
- 所有改变状态的 Cookie API 继续受 `SameSite` 与同源检查保护。
- 日志不得包含用户输入的密码、Cookie、会话令牌或环境变量内容。

## 5. 家庭数据共享

### 5.1 共享范围

以下内容继续由服务器 SQLite 统一持久化，因此两台手机共享：

- 菜谱及其食材、调料、步骤、标签和图片 URL
- 收藏状态
- 做菜次数
- 复盘内容与评分
- 已完成的导入结果
- AI 食材图片缓存

不新增 `users`、`households` 或 `household_id`。当前部署实例本身代表唯一家庭数据空间。

### 5.2 设备本地范围

以下临时状态继续保留在当前手机：

- 正在做菜的当前步骤和已完成步骤
- 计时器和语音播报状态
- 备料勾选
- 未完成的导入草稿
- 列表返回位置和临时筛选界面状态

这样可以避免两台手机同时操作时互相跳步骤、暂停计时或覆盖未完成草稿。

### 5.3 刷新策略

不引入实时推送。另一台手机获取新数据的时机为：

- 首次进入页面
- 路由重新进入
- 浏览器从后台回到前台时，对当前菜谱列表或详情执行一次轻量刷新

刷新失败保留页面现有数据，并提供非阻塞提示；不得清空已经渲染的菜谱。

## 6. 食材图生成状态与压缩

### 6.1 自动生成时机

- 进入做菜模式后立即渲染食材卡片，不等待 AI。
- 即将进入横向食材轨道可视区域的卡片自动请求图片。
- 相同规范化食材名称继续共用缓存；已有缓存时直接返回，不重复调用 AI。
- 不在菜谱导入阶段同步生图，避免拖慢主流程。

### 6.2 骨架反馈

- 复用现有 shadcn/ui `Skeleton`，不新造基础组件。
- 加载中保留食材首字作为可识别占位，并增加柔和的圆形加载环。
- 容器暴露 `aria-busy=true` 和屏幕阅读器可读的“正在生成食材图片”。
- 图片完成后使用约 160–200ms 淡入替换，不弹跳、不推动布局。
- `prefers-reduced-motion` 下取消旋转和缩放，只保留静态骨架与即时替换。
- 请求失败时结束骨架，恢复首字占位；做菜步骤、计时和语音不受影响。

### 6.3 小图优化

Micu 请求继续使用 `quality=low`。由于上游仍可能返回 1–3MB 的大尺寸 PNG，服务端在落缓存前执行一次头像级优化：

- 以圆形卡片用途为目标裁切到 256×256。
- 通过项目直接依赖的 `sharp` 裁切，并以质量 60 编码为 WebP；不依赖 Next.js 的传递依赖。
- 保留足够覆盖 72px CSS 尺寸在高像素密度手机上的清晰度。
- 缓存版本升级为 `v2`，文件键继续由提示词版本和规范化食材名派生。
- 图片读取接口返回正确 `image/webp`、稳定 ETag 和长缓存头。

此优化减少网络传输和浏览器解码时间，但不会缩短首次上游 AI 推理本身。首次等待通过骨架反馈解释，后续相同食材通过缓存快速出现。

### 6.4 错误恢复

- AI 超时、限流或服务错误：不写缓存，客户端恢复首字；下次重新进入可再次尝试。
- 下载内容超过上限、格式非法或压缩失败：不落盘原图。
- 缓存文件损坏：删除损坏文件并允许重新生成。
- 同一个缓存键的并发请求继续共享服务器内同一个生成 Promise。
- 客户端卸载时取消等待；服务端已经开始的安全生成可以完成并写入缓存。

## 7. 备料全选交互

- 继续使用 shadcn/ui `Button` 和既有 Apple 按压反馈。
- 未全部选中时显示“全部勾选”，点击后选中当前全部食材。
- 全部选中时显示“取消全选”，再次点击后清空全部选中状态。
- 部分选中时显示“全部勾选”，点击补齐全部选中。
- 按钮提供与状态一致的 `aria-pressed`。
- 食材单项仍可独立点击切换，且不得影响步骤完成计数。

## 8. 备份与恢复

- 每日备份 SQLite 主库以及 WAL 一致性检查后的数据库快照，保留最近 7 份。
- 每周保留 1 份长期备份，保留最近 4 份。
- 应用升级、数据库迁移或切换镜像前额外创建一次备份。
- 生成图片目录每周备份一次并保留最近 4 份，避免恢复后重新消耗 AI 生图额度。
- 恢复演练至少验证：菜谱数量、最近复盘、收藏状态和一张食材缓存图。
- 部署文档提供备份、恢复、升级和回滚命令，但实际服务器操作需用户另行授权。

## 9. 组件与文件边界

预计新增或修改以下职责边界：

- `src/middleware.ts`：页面和 API 会话门禁。
- `src/lib/auth/`：密码摘要、会话签名和纯验证逻辑。
- `src/app/unlock/`：家庭解锁页面。
- `src/app/api/auth/`：登录、退出和会话相关接口。
- `src/components/`：退出家庭入口与前台恢复刷新。
- `src/lib/db/client.ts`：SQLite WAL、外键和忙等待初始化。
- `src/components/cooking/ingredient-rail.tsx`：图片骨架状态和全选切换。
- `src/lib/images/`：256px WebP 优化、v2 缓存与内容校验。
- `src/app/api/ingredient-images/[key]/`：返回 WebP 和缓存头。
- `scripts/`：家庭密码摘要生成与部署备份辅助脚本。
- 部署文档：环境变量、持久化目录、反向代理、备份与恢复说明。

所有常用交互继续复用 shadcn/ui 原语。不得用只在前端隐藏内容的方式替代服务端门禁，也不得把原生按钮包装成新的重复基础组件。

## 10. 精简测试与验收

采用先失败测试、后实现的方式，但限制重复验证：

### 10.1 针对性测试

- 正确密码可以建立会话，错误密码和畸形输入被拒绝。
- 密码摘要比较、会话签名、过期和篡改验证。
- 未登录页面重定向、未登录 API 返回 `401`、静态资源与解锁页可访问。
- 已登录会话可访问菜谱 API，退出后立即失效。
- SQLite 初始化启用外键、WAL 和忙等待。
- 食材图加载期间显示骨架，成功后替换，失败后回退首字。
- 原始大图被压缩成 256×256 WebP，缓存命中不再次调用 AI。
- “全部勾选”再次点击变成取消全选。

### 10.2 最终验证

仅在针对性测试稳定后执行一次：

1. 相关测试集。
2. lint 与生产构建。
3. 完整测试集。
4. 一条 390px 手机流程：解锁 → 菜谱列表 → 菜谱详情 → 做菜模式 → 骨架 → 图片出现 → 全选 → 取消全选 → 退出家庭。
5. 一次数据库重启持久性检查，以及一次备份文件可读性检查。

不重复跑多尺寸全页面截图矩阵。验收报告只记录最终有效证据与未覆盖的真实服务器部署边界。

## 11. 验收标准

- 未解锁访问者无法读取或修改任何家庭业务数据。
- 正确家庭密码建立 30 天安全会话，错误密码不泄露内部信息。
- 密码、摘要输入、Cookie、会话密钥和 AI Key 不进入客户端、Git 或日志。
- 两台手机访问同一云端实例时，共享菜谱、收藏、做菜次数和复盘。
- 做菜中临时状态仍只影响当前手机。
- SQLite 和食材图片位于持久化目录，重启应用后数据保持。
- 页面重新进入或从后台恢复时能获取另一台手机已经提交的新数据。
- 食材图生成期间有清晰但克制的骨架反馈，失败不阻塞做菜。
- 缓存食材图为约 256×256 WebP，不再向手机传输 1–3MB 原始 PNG。
- “全部勾选”可再次点击取消全部选择。
- 390px 主流程无横向溢出，加载动效尊重减少动态效果设置。

## 12. 后续升级触发条件

出现以下任一需求时，再单独设计中心数据库迁移：

- 多个家庭使用同一部署。
- 老公和老婆需要独立身份、个人收藏或操作记录。
- 多实例水平扩容。
- 需要跨设备实时接力做菜或协同编辑。
- 需要将图片统一迁入对象存储。

在这些条件出现前，单实例持久化 SQLite 是复杂度最低且满足当前家庭使用目标的方案。
