# 家庭密码统一与菜谱列表下拉刷新设计

日期：2026-07-20
状态：对话方案已确认，等待书面规格复核

## 目标

1. 当前本地家庭服务与未来云部署统一使用用户在本次对话指定的家庭密码；规格和仓库不重复记录其明文。
2. 菜谱列表提供符合移动端习惯的下拉刷新，不整页重载，不隐藏已经展示的菜谱。
3. 复用现有列表后台加载、竞态取消和失败保留机制，不新增第二套数据同步逻辑。

## 非目标

- 本轮不部署或修改云服务器。
- 不实现 WebSocket、轮询或跨设备实时推送；另一台手机通过下拉刷新、重新聚焦或重新进入页面获取最新数据。
- 不改变菜谱 API、SQLite schema、家庭账号模型或公开部署阻塞结论。
- 不在仓库、报告、日志或截图中保存明文密码、会话密钥或真实 AI key。

## 1. 家庭密码

- 当前本地演示服务重启时，在进程环境中注入目标密码的 scrypt hash；不修改或读取现有 `.env` 中的真实密钥。
- 更换密码时同步轮换 `FAMILY_SESSION_SECRET`，使两台手机上已有 Cookie 立即失效；用户需要用新密码重新解锁。
- 未来云部署通过现有 `npm run auth:hash` 安全生成同一密码的 hash，并写入服务器上权限为 `0600` 的私有环境文件；仓库只保留无值模板和操作说明。
- 继续沿用现有登录限流、通用错误提示、HttpOnly Cookie 与同源校验。
- 用户指定的目标密码强度较低，因此本轮仍不改变“公开部署未就绪”的结论；正式公网开放前应由家庭确认是否换成更强口令。

## 2. 下拉刷新交互

### 2.1 触发范围

- 只集成到现有 `RecipeList`，因此 `/recipes` 及复用该组件的分类列表行为一致。
- 仅当页面位于顶部、当前没有初始加载且没有进行中的下拉刷新时，向下的纵向手势才可进入下拉刷新。
- 10px 的方向判定迟滞后才接管手势；横向滑动筛选标签、普通向上滚动和非顶部拖动不触发刷新。
- 手势一旦进入下拉刷新，取消菜谱卡片的长按计时，避免误入管理模式。

### 2.2 物理反馈

- 内容表面跟随手指向下移动，超出边界后使用 Apple rubber-band 公式、当前 viewport 高度和 `0.55` 阻力常数；视觉位移上限为 96px，避免拖出大面积空白。
- 原始纵向拖动未达到 64px 提交阈值时显示“下拉刷新”；达到阈值后显示“松开刷新”。
- 未达阈值松手：从当前显示位置以临界阻尼弹簧回到原位，不触发网络请求。
- 达到阈值松手：固定在 44px 的紧凑刷新位置，显示旋转图标和“正在同步”；同一时刻最多一个刷新请求。
- 成功后显示勾选图标和“已更新”450ms，随后回弹；失败时显示失败状态，并沿用现有 toast“同步失败，已保留当前菜谱”。
- 动画从当前可见位置继续，允许用户在提交前反向拖回；回弹使用 0.35s、无 overshoot 的临界阻尼 spring，不使用不可中断的关键帧位移动画。

### 2.3 数据行为

- 下拉动作调用现有 `load({ background: true })`。
- 已展示的菜谱在请求期间保持可见，不出现初始 Skeleton，也不清空筛选、搜索、滚动恢复或管理选择。
- `load` 返回内部成功/失败结果供刷新指示器显示准确的完成状态；现有调用者可以忽略该返回值。
- 新请求继续中止旧请求并忽略迟到响应；下拉刷新不得削弱现有 foreground refresh 的竞态保护。
- 成功后以服务器返回的当前筛选结果替换列表；失败后保留最后一次成功内容。

## 3. 组件与现有组件库

- 新增一个小型、列表专用的下拉刷新 hook/视图边界，手势状态与菜谱数据加载保持解耦。
- 使用项目已安装的 Framer Motion motion value/spring 实现手指跟随和回弹，不引入新动画依赖。
- 指示图标使用 Lucide；“更多”菜单增加 shadcn `DropdownMenuItem`“刷新菜谱”，作为桌面端、键盘和辅助技术的等价入口。
- 不重新实现按钮、菜单、toast 或 loading primitive。

## 4. 可访问性与降级

- 指示器使用 `role="status"`、`aria-live="polite"`；刷新期间列表容器暴露 `aria-busy="true"`。
- 列表页面挂载期间抑制浏览器原生 pull-to-refresh；组件卸载时恢复，不影响其他页面。
- `prefers-reduced-motion: reduce` 时取消内容大幅位移和弹性回弹，只保留紧凑的透明度/状态变化。
- 触摸手势不可用时仍可通过“更多 → 刷新菜谱”完成同一后台刷新。
- 刷新失败不会用错误页覆盖已有内容，读屏状态与 toast 都使用不包含服务器内部信息的中文提示。

## 5. 测试与验收

严格执行 RED → GREEN：先写失败测试并确认失败原因是功能缺失，再写实现。

定向自动化覆盖：

1. 非顶部拖动不刷新。
2. 小于阈值松手只回弹，不调用 API。
3. 超过阈值只触发一次后台请求，旧卡片与筛选保持可见且无 Skeleton。
4. 刷新期间重复手势不会产生第二个请求。
5. 成功更新列表；失败保留旧列表并显示现有同步失败提示。
6. 手势进入后取消长按管理计时。
7. 指示器、`aria-busy` 与 shadcn 菜单刷新入口可访问。
8. reduced-motion 分支不执行大幅内容位移。

执行范围：相关 hook/组件测试、`recipe-list-v3.test.tsx`、一次 lint/diff check，以及一次 390px 手机流程检查；不重复完整项目 E2E 或真实 AI 测试。

## 6. 验收标准

- 本地服务重启后，旧登录态失效；两台手机均只能用用户指定的新密码解锁。
- 手机在列表顶部下拉超过阈值并松手，可看到连续、克制的状态反馈和服务器最新菜谱。
- 手机 A 新增菜谱后，手机 B 在列表下拉刷新即可看到；失败不会丢失手机 B 现有列表。
- 普通滚动、横向筛选、卡片点击和长按管理没有回归。
- 仓库不包含家庭密码明文配置、真实 session secret 或真实 API key。
